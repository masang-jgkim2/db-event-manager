# 검토: 이벤트 1건에 여러 서버·여러 쿼리 실행

> 작성일: 2026-03-10  
> 목적: "한 이벤트에서 여러 서버에 대해 여러 쿼리를 실행해야 하는" 요구와 현재 구조의 차이 검토

---

## 1. 현재 구조 요약

### 1.1 데이터 모델

| 개념 | 현재 구조 | 비고 |
|------|-----------|------|
| **이벤트 인스턴스** | 1건당 **프로덕트 1개**(nProductId), **쿼리 1개**(strGeneratedQuery) | 여러 서버/여러 쿼리 미지원 |
| **DB 접속 정보** | (nProductId, strEnv) 당 **1건** | 프로덕트·환경별 단일 접속. 여러 서버 = 여러 프로덕트 또는 스키마 확장 필요 |
| **실행** | (인스턴스, strEnv) → **접속 1개** 조회 → **strGeneratedQuery 1개** 실행 | 한 연결에서 세미콜론 분리 멀티 문장은 지원 |

### 1.2 실행 흐름 (한 번의 Execute)

```
이벤트 인스턴스 1건
  → nProductId 1개
  → fnFindActiveConnection(nProductId, strEnv) → DB 접속 정보 1건
  → fnExecuteQueryWithText(objDbConn, strGeneratedQuery, strEnv)
     → 같은 연결 위에서 fnParseQueries(세미콜론 분리) 후 문장별 실행·트랜잭션
```

- **한 이벤트 = 한 프로덕트 = 한 접속(서버) = 한 “쿼리 문자열”(내부적으로 여러 문장 가능)** 이다.
- 따라서 **이벤트 1건으로 “서버 A + 서버 B + …” 처럼 여러 서버에 나눠 실행하는 구조는 아니다.**

---

## 2. 요구 시나리오와의 차이

### 2.1 요구: 이벤트 1개 → 여러 서버 × 여러 쿼리

예시:

- 서버 1 (예: 게임 DB A): `UPDATE ...` , `INSERT ...`
- 서버 2 (예: 게임 DB B): `DELETE ...`
- 서버 3 (예: 로그 DB): `INSERT INTO audit ...`

또는 같은 프로덕트라도 **물리적으로 다른 DB**(리전/샤드)에 같은 이벤트를 반영하는 경우.

### 2.2 현재 구조로 가능한 것 / 불가능한 것

| 요구 | 현재 구조 | 비고 |
|------|-----------|------|
| 한 서버에 **여러 쿼리**(문장) | ✅ 가능 | strGeneratedQuery 에 세미콜론으로 여러 문장 작성 → fnParseQueries 로 분리 후 한 트랜잭션으로 실행 |
| **여러 서버**에 각각 쿼리 실행 | ❌ 불가 | 인스턴스는 nProductId 1개, fnFindActiveConnection 도 1건만 반환. 서버별 접속·쿼리 분리 없음 |
| 이벤트 1건으로 “서버 A에 쿼리1, 서버 B에 쿼리2” | ❌ 불가 | 쿼리 1개·접속 1개 전제 |

---

## 3. 구조가 맞지 않는 지점 정리

1. **인스턴스가 “한 프로덕트·한 쿼리”에 묶여 있음**  
   - nProductId 1개, strGeneratedQuery 1개만 저장.  
   - “대상 서버(접속) 목록”과 “서버별 쿼리”를 표현할 수 없음.

2. **접속 정보가 (프로덕트, 환경)당 1건**  
   - fnFindActiveConnection(nProductId, strEnv) → 1건.  
   - 같은 프로덕트라도 서버가 여러 대(리전/샤드)이면, 현재 스키마만으로는 “여러 접속”을 이벤트와 연결하기 어렵다.

3. **실행 API가 “한 환경에 한 번만” 실행**  
   - POST execute 시 strEnv(qa/live) 하나만 받고, 한 접속·한 쿼리 문자열만 실행.  
   - “여러 접속에 순서대로/병렬로 실행”하는 단계가 없음.

---

## 4. 확장 방향 제안

요구를 “**이벤트 1건 = 여러 (서버/접속, 쿼리) 조합**”으로 끌어오려면, 아래 중 한 축을 확장해야 한다.

### 4.1 방안 A: 인스턴스에 “실행 대상 목록” 추가

- **모델 확장**  
  - 인스턴스에 **실행 대상 배열**을 둠.  
    - 예: `arrExecutionTargets: Array<{ nDbConnectionId: number; strQuery: string }>`  
    - 또는 `nDbConnectionId` 대신 `nProductId, strEnv, strServerKey?` 같은 식으로 “어떤 접속인지” 식별.
- **실행 로직**  
  - Execute 시 `arrExecutionTargets` 를 순회하며,  
    - 접속 ID(또는 product+env+key)로 DB 접속 정보 1건씩 조회  
    - 해당 `strQuery` 만 fnParseQueries 후 fnExecuteQueryWithText 로 실행  
  - “전부 성공 시에만 상태 전이” vs “일부 실패 시 롤백 정책”은 정책에 따라 결정 (순차 실행 시 이전 서버 롤백 불가하므로, 보통은 “실패 시 중단 + 이미 실행된 건은 그대로” 또는 “전부 성공할 때만 성공 처리” 정도).
- **기존 단일 쿼리와의 호환**  
  - `arrExecutionTargets` 가 비어 있으면 기존처럼 `nProductId + strGeneratedQuery` 1건으로 실행하도록 fallback 하면, 기존 이벤트와 호환 가능.

### 4.2 방안 B: 프로덕트·환경당 “접속 여러 개” 허용

- **모델 확장**  
  - DB 접속 정보에 “동일 (nProductId, strEnv) 내에서 구분자” 추가.  
    - 예: `strServerKey` (예: "kr", "global", "log") 또는 `nOrder`.  
  - 제약 완화: (nProductId, strEnv) 당 1건 → (nProductId, strEnv, strServerKey) 당 1건 등으로 변경.
- **인스턴스**  
  - “어느 서버에 어떤 쿼리”를 저장하려면, 여전히 **인스턴스 쪽에 “대상 목록 + 쿼리”**가 필요 (방안 A와 동일한 개념).  
  - 즉, 접속이 여러 개만 있어도 “이벤트 1건이 그 중 여러 개를 선택”하려면 인스턴스 스키마 확장이 필요.

### 4.3 방안 C: 이벤트 1건을 “여러 인스턴스”로 분리 (구조 변경 최소)

- **운영 방식**  
  - “한 이벤트”를 비즈니스 단위로만 두고, **실제 실행 단위는 “서버 1개당 인스턴스 1건”**으로 둠.  
  - 예: 이벤트 “2월 정기 점검” → 인스턴스 A(프로덕트1, 쿼리1), 인스턴스 B(프로덕트2, 쿼리2) 이렇게 2건 생성.  
  - UI/워크플로에서 “한 그룹”으로 묶어서 보여주고, 실행은 각 인스턴스별로 기존 Execute API 그대로 호출.
- **특징**  
  - 현재 모델·API를 거의 그대로 쓸 수 있음.  
  - 대신 “한 번의 클릭으로 여러 서버 동시 실행”이 아니라 “인스턴스별로 실행”이 되고, 트랜잭션은 서버 단위로만 유지됨.

---

## 5. 정리

| 항목 | 내용 |
|------|------|
| **현재 구조** | 이벤트 1건 = 프로덕트 1개 = DB 접속 1개 = 쿼리 1개(멀티 문장 가능). **여러 서버에 여러 쿼리**는 미지원. |
| **갭** | 인스턴스가 “한 접속·한 쿼리”에 묶여 있음. 접속도 (프로덕트, 환경)당 1건 전제. |
| **확장 시** | 인스턴스에 “실행 대상 목록”(접속 또는 product+env+key + 쿼리)을 두고, Execute 시 그 목록을 순회해 접속별로 쿼리 실행하는 방식(방안 A)이 요구와 가장 잘 맞음. |
| **최소 변경** | 구조는 그대로 두고, “한 이벤트”를 서버별 인스턴스 여러 건으로 나누어 운영(방안 C)하는 방법도 있음. |

원하시면 방안 A 기준으로 **스키마/인터페이스 초안**(예: `arrExecutionTargets` 필드 스펙, Execute API 시그니처 변경)까지 구체화해 드리겠습니다.
