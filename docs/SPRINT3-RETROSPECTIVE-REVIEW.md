## A. 도구 도입 정당성 / UX (업무 효율 vs 불편)

### 항목
- _알로 링크 vs DB 프로세스 매니저 링크_
- _업무 프로세스 뎁스 하나 더 늘어난 거 아닌가? (사업 기획 → 알로 카드 → DQPM 링크)_
- _툴을 사용하는 이유가 편하기 위함인데 더 불편한 거?_

### 검토 결과
- **게스트 권한 불필요** — 쿼리 내용·반영 날짜·DB 구조가 포함된 내부 툴이므로 로그인 필수 유지가 맞음.
- 필요한 것은 **딥링크 + 로그인 후 자동 복귀** 패턴: 알로에 `/my-dashboard?nId=123` 링크 → 로그인 안 됐으면 `/login?redirect=/my-dashboard?nId=123` → 로그인 후 상세 자동 열림.
- 링크 복사 버튼을 각 이벤트 행에 추가하면 알로 첨부 마찰 제거.

### 대응 방안
| 상태 | 방안 | 난이도 |
|------|------|--------|
| ✅ 구현 완료 | **이벤트 인스턴스 퍼머링크** (`/my-dashboard?nId=N`) + 로그인 후 자동 상세 열림 + **링크 복사 버튼** | 낮~중 |
| 보류 | 툴 도입 이점 인트로/온보딩 문서 | 낮음 |
| 장기 | 알로 Webhook/API 양방향 연동 | 높음 |

## B. 데이터 안전·무결성

### 항목
- _DB 반영 바로 되는 건가? 백업 됐냐?_
- _쿼리 반영 전후, 스키마 변경 관리_

### 검토 결과
- **DQPM에서 직접 차등 백업 실행은 현실적이지 않음** — DB 서버 백업 권한 필요, 대용량 DB 소요 시간 불명확, 실패 처리 복잡.
- 현실적 대안: **실행 전 확인 팝업에 체크박스** ("백업 완료 여부 확인했습니다") — 책임 소재도 명확해짐.
- 차등 백업 자동화는 DB 인프라팀과 별도 협의 후 **장기 과제**로 분리.

### 현재 상태
- **반영 날짜 검증** 있음 (DEV/QA: 반영일 이전, LIVE: 반영일 이후만 실행).
- **트랜잭션** 실행 (MSSQL `BEGIN TRAN`, MySQL).
- **백업 여부·반영 전후 스냅샷** 기록 없음.

### 대응 방안
| 상태 | 방안 | 난이도 |
|------|------|--------|
| 4차 예정 | 실행 전 **백업 확인 체크박스 팝업** (백업 직접 실행 X, 사람이 확인) | 낮음 |
| 검토 중 | 실행 이력에 "실행 전 쿼리 전문" 저장 | 중 |
| 장기 | DB 인프라팀과 차등 백업 연동 협의 | 높음 |

## C. 변경 인지 / 협업 (DBA·서버팀·사업팀)

### 항목
- _기능이나 테이블의 변경점을 인지 못할 경우가 있다_
- _새로 작성된 쿼리를 DBA가 모를 수 있다? 새로운 테이블이나 기능은 깃랩_
- _실제 쿼리 리뷰 업무를 녹일 수 있는 방법_
- _서버팀에서 실행? 서버팀 요청 → 권한 부여하면 가능_
- _사업팀에서 사용 가능한 거? 협의가 됐냐?_

### 검토 결과
- **서버팀 실행**: RBAC로 이미 가능 — 서버팀 역할 생성 + `my_dashboard.execute_live` 등 권한 부여하면 됨. **기능 개발 불필요, 협의 후 역할 추가만 하면 됨.**
- **사업팀 접근**: 읽기 전용(이벤트 상태 확인)인지, 이벤트 생성까지인지 먼저 협의 필요. 협의 후 역할 생성.
- **출조·라그 신규 쿼리**: 프로덕트 추가 → DB 접속 → 쿼리 템플릿 등록 흐름으로 이미 가능. **온보딩 가이드 문서만 없는 상태.**
- **쿼리 리뷰 단계**: 별도 워크플로 단계 추가보다 **컨펌 단계에 리뷰 코멘트 필드 추가** 가 현실적 (워크플로 변경 최소화).
- **변경 인지 문제**: 신규 쿼리 플래그보다 **영향 테이블 명시 필드** 추가가 더 정확한 해결책.

### 대응 방안
| 상태 | 방안 | 난이도 |
|------|------|--------|
| 협의 필요 | 서버팀 역할 권한 정의 → 역할 추가 (기능 개발 없음) | 낮음 |
| 협의 필요 | 사업팀 접근 범위 정의 → 역할 추가 | 낮음 |
| 4차 검토 | 이벤트 생성 폼에 **영향 테이블 명시 필드** 추가 | 낮음 |
| 4차 검토 | 컨펌 단계에 **리뷰 코멘트 필드** 추가 | 중 |
| 문서화 | 신규 서비스 온보딩 체크리스트 (프로덕트 → DB → 템플릿 → 권한) | 낮음 |
| 장기 | 깃랩 MR/이슈와 DQPM 이벤트 양방향 연결 | 높음 |

## D. 날짜·환경 관리

### 항목
- _QA 반영 날짜, LIVE 반영 날짜_
- _출조·라그 신규 쿼리는 어떻게?_
- _신규 쿼리?_

### 검토 결과
- **QA/LIVE 날짜 분리**는 스키마(`dtQaDeployDate`, `dtLiveDeployDate`)·UI·검증 로직 동시 변경 필요 → **설계 먼저, 4차 이후 구현**.
- 신규 서비스(출조·라그)는 기능적으로 이미 가능 — **온보딩 가이드 문서 부재** 가 핵심.

### 현재 상태
- 이벤트 인스턴스에 **`dtDeployDate`** 1개 — QA/LIVE 날짜 구분 없음.

### 대응 방안
| 상태 | 방안 | 난이도 |
|------|------|--------|
| 설계 필요 | **QA / LIVE 반영 날짜 분리** (`dtQaDeployDate`, `dtLiveDeployDate`) | 높음 |
| 임시 | 설명 필드에 QA/LIVE 날짜 기재 가이드 + 현행 `dtDeployDate` 유지 | 낮음 |
| 문서화 | 신규 서비스 온보딩 가이드 | 낮음 |

## E. 테스트·검증

### 항목
- _비교 분석 테스트를 할 수 있으면 좋을 듯_

### 검토 결과
- 완전한 SELECT 전후 비교는 복잡도 높음 → 단계적 접근.
- 1단계: DEV/QA/LIVE 실행별 **영향 행 수를 세트별로 저장·표시** (이미 일부 있음).
- 2단계: 실행 전 SELECT 미리보기 (장기).

### 대응 방안
| 상태 | 방안 | 난이도 |
|------|------|--------|
| 4차 검토 | 실행 결과(영향 행 수)를 환경별로 저장·비교 표시 | 중 |
| 장기 | 실행 직전 SELECT 미리보기 (영향 행 조회) | 높음 |
