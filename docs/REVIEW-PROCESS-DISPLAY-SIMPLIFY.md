# 보이는 프로세스 간소화 검토

권한은 그대로 두고, **나의 대시보드에서 보이는 프로세스**만 아래 구조로 간소화하는 경우의 매핑·영향 검토입니다.

---

## 1. 제안하신 표시 구조

```
생성
  → 컨펌 (요청, 컨펌중, 완료)
  → QA (쿼리 반영 요청, 쿼리 반영 완료, 쿼리 반영 확인, 재요청)
  → LIVE (쿼리 반영 요청, 쿼리 반영 완료, 쿼리 반영 확인, 재요청)
  → 완료
```

- **권한**: 변경 없음 (기존 권한 코드·역할 권한 화면 그대로).
- **변경 대상**: 스테퍼/상태 태그 등 **표시용 라벨·그룹**만.

---

## 2. 현재 상태 코드 ↔ 간소화 표시 매핑

| 현재 상태 (코드) | 제안 표시 (간소화) | 비고 |
|------------------|--------------------|------|
| event_created | **생성** | 그대로 1:1 |
| confirm_requested | 컨펌 **요청** | 컨펌 단계 1번째 |
| (동일 상태 대기) | 컨펌 **컨펌중** | 요청 후 DBA 대기 = 같은 confirm_requested를 "컨펌중"으로 표시할지 여부 (선택) |
| dba_confirmed | 컨펌 **완료** | 컨펌 단계 마지막 |
| qa_requested | QA **쿼리 반영 요청** | QA 1번째 |
| qa_deployed | QA **쿼리 반영 완료** | QA 2번째 |
| qa_verified | QA **쿼리 반영 확인** | QA 3번째 |
| (qa_verified → qa_requested 전이) | QA **재요청** | 액션이지 상태 아님. 스텝으로는 “확인” 다음 선택지로 표시 가능 |
| live_requested | LIVE **쿼리 반영 요청** | LIVE 1번째 |
| live_deployed | LIVE **쿼리 반영 완료** | LIVE 2번째 |
| live_verified | LIVE **쿼리 반영 확인** = **완료** | LIVE 3번째이자 전체 프로세스 **완료** |

---

## 3. 컨펌 단계 “컨펌중” 처리

- **현재**: `confirm_requested` 하나만 있음 (요청됨 = DBA 대기).
- **제안**: 컨펌을 `요청 | 컨펌중 | 완료` 3단계로 보이게 함.
- **구현 옵션**
  - **A)** `요청` = confirm_requested, `완료` = dba_confirmed만 쓰고, **컨펌중**은 빼서 2단계로 표시 (요청, 완료).
  - **B)** 3단계 유지: `요청`(과거), `컨펌중`(current = confirm_requested), `완료`(current = dba_confirmed). 즉 confirm_requested일 때 두 번째 칸을 “컨펌중”으로 표시.

제안문서대로라면 **B**가 요청 의도에 가깝습니다.

---

## 4. QA / LIVE “재요청”

- **재요청**은 상태가 아니라 **전이**(qa_verified → qa_requested, live_deployed/live_verified → live_requested).
- 스테퍼에서는 “쿼리 반영 확인” 다음에 **재요청 시 다시 앞 단계로 돌아감**으로 표현하면 됨.
- 별도 “재요청” 스텝을 두지 않아도 되고, 재요청 시 스텝 인덱스만 다시 요청/완료 쪽으로 이동하면 됨.

---

## 5. 적용 시 수정 포인트 (코드)

| 위치 | 내용 |
|------|------|
| **표시용 라벨** | `OBJ_STATUS_CONFIG`(또는 스테퍼 전용 라벨 맵)를 위 간소화 표시명으로 변경. |
| **스테퍼 구조** | `fnBuildSteps`를 **5개 페이즈**로 재구성: 생성 → 컨펌(2~3 서브스텝) → QA(3 서브스텝) → LIVE(3 서브스텝) → 완료. |
| **현재 단계 계산** | `objInstance.strStatus` → 해당하는 **페이즈 + 페이즈 내 인덱스**로 매핑하는 함수 추가. |
| **테이블/상세 상태 태그** | 지금처럼 `OBJ_STATUS_CONFIG[strStatus].strLabel`을 쓰는 곳은, 같은 매핑으로 “간소화 표시” 문자열을 쓰면 됨. |

---

## 6. QA 스킵(단일 서버 LIVE만) 시

- `arrDeployScope === ['live']` 이면 QA 단계 없이: 생성 → 컨펌 → LIVE → 완료.
- 간소화 구조에서도 **QA 블록을 비표시**로 두면 되고, 기존 로직과 동일하게 처리 가능.

---

## 7. 간단히 보기 / 자세히 보기 옵션

**한쪽으로만 고정하지 않고**, 사용자가 선택할 수 있게 두는 방식입니다.

| 옵션 | 표시 방식 | 용도 |
|------|-----------|------|
| **간단히 보기** | 5단계(생성 → 컨펌 → QA → LIVE → 완료) + 페이즈별 서브스텝 | 전체 흐름만 빠르게 보고 싶을 때 |
| **자세히 보기** | 현재처럼 상태별 9스텝(생성, 컨펌 요청, DBA 컨펌 완료, QA 반영 요청, …) | 단계·권한명을 하나하나 보고 싶을 때 |

- **저장**: 설정 Drawer(굳굳 설정) 또는 나의 대시보드 상단에 스위치/라디오로 두고, `localStorage` 또는 기존 테마/설정 스토어에 `bProcessSimpleView` 같은 값으로 저장.
- **기본값**: 간단히 보기 / 자세히 보기 중 하나를 기본으로 두면 됨 (예: 기본은 간단히 보기).
- **적용 범위**: 스테퍼(행 확장), 테이블 상태 컬럼·태그, 상세 모달의 상태 표시에 동일 옵션 적용.

이렇게 하면 “항상 간소화” vs “항상 자세히”를 사용자 선택으로 바꿀 수 있어서, 검토 시에는 자세히, 일상 사용에는 간단히 보기로 쓰기 좋습니다.

---

## 8. 정리

- **권한·백엔드 상태값**: 그대로 두고, **보이는 프로세스만** 간소화 구조로 추가.
- **간단히 보기 / 자세히 보기** 옵션을 두면, 한쪽만 쓰는 것보다 유연하고 사용성 좋음.
- 컨펌은 “요청 / 컨펌중 / 완료” 3단계로 할지, “요청 / 완료” 2단계로 할지만 선택하면 됨.
- 재요청은 별도 스텝 없이, “확인” 이후 전이로만 반영해도 됨.

원하시면 다음 단계로 옵션 플래그 + `fnBuildSteps` 분기(간단/자세히)용 코드 초안을 적어드리겠습니다.
