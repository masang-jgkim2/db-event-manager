# 검토: 메뉴 위치·보드/테이블 크기 사용자 지정

> 작성일: 2026-03-10  
> 목적: 현재 고정 UI에서 사용자가 **메뉴 위치**와 **보드/테이블 크기**를 마음대로 바꿀 수 있게 하는지 검토

---

## 1. 현재 이미 지원하는 사용자 지정

| 항목 | 구현 위치 | 저장 | 비고 |
|------|-----------|------|------|
| **사이드바 너비** | MainLayout.tsx (드래그 리사이즈) | useThemeStore → localStorage `db-event-manager-theme` | 최소 160px, 최대 화면 1/3 |
| **테이블 컬럼 순서** | AppTable.tsx (dnd-kit 드래그) | localStorage `app_table_col_order_{strTableId}` | 테이블별로 컬럼 순서 유지 |
| **테이블 컬럼 너비** | AppTable.tsx (헤더 셀 드래그 리사이즈) | localStorage `app_table_col_width_{strTableId}` | 테이블별로 컬럼 너비 유지 |

- 테마(다크/라이트, 폰트 크기, 컴팩트, primary 색 등)도 useThemeStore로 localStorage에 저장됨.

---

## 2. 메뉴 위치 사용자 지정

### 2.1 요구사항 정리

- **그룹 순서**: "이벤트" / "사용자" / "운영" 순서를 사용자가 바꿀 수 있음 (예: 사용자 → 이벤트 → 운영).
- **그룹 내 항목 순서**: 같은 그룹 안에서도 메뉴 항목 순서 변경 (예: "나의 대시보드"를 "이벤트 생성"보다 위로).

### 2.2 기술적 가능 여부

- **가능함.** 메뉴 데이터는 MainLayout에서 `arrMenuItems`로 생성되므로, "표준 순서"를 정의해 두고 사용자 지정 순서만 저장했다가 적용하면 됨.

### 2.3 구현 방식 제안

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 설정에서 순서 변경** | UI 설정 드로어(또는 전용 모달)에 "메뉴 순서" 섹션 추가. 그룹/항목을 리스트로 보여주고 위아래 화살표 또는 드래그로 순서 변경 후 저장. | 사이드바 구조 유지, 권한에 따라 안 보이는 항목은 리스트에서 제외 가능. | 메뉴를 보면서 바로 조작하는 느낌은 약함. |
| **B. 사이드바에서 드래그** | 사이드바 메뉴 항목을 드래그해서 순서 변경. | 직관적, WYSIWYG. | Ant Design Menu는 기본이 드래그 미지원 → 커스텀 렌더링 또는 별도 Sortable 리스트 필요. 그룹/항목 계층 처리 필요. |

- **권장**: 1단계는 **방식 A**(설정에서 순서 변경). 저장 구조만 맞추면 나중에 방식 B로 확장 가능.

### 2.4 저장 구조 제안

- **저장소**: useThemeStore 확장 또는 별도 `useLayoutStore` (persist).
- **저장 데이터 예시**:
  - `arrMenuGroupOrder: string[]` — 그룹 key 순서, 예: `['user-group', 'event-group', 'operation-group']`.
  - `arrMenuItemOrderByGroup: Record<string, string[]>` — 그룹별 메뉴 key 순서, 예: `{ 'event-group': ['/', '/products', '/events', '/db-connections'] }`.
- **렌더링**: MainLayout에서 `arrMenuItems`를 만든 뒤, 위 순서에 따라 그룹 정렬·그룹 내 children 정렬한 배열을 실제 메뉴에 전달.
- **기본값**: 현재 코드 순서(이벤트 → 사용자 → 운영)를 기본으로 두고, 사용자가 한 번이라도 변경한 경우에만 저장값 사용.

### 2.5 주의사항

- **권한**: 권한이 없어서 안 보이는 메뉴는 순서 리스트에서 제외하거나, 리스트에는 있되 클릭 시 403 등으로 처리. 저장된 순서에 없는 새 메뉴(코드 추가분)는 기본 위치에 삽입하는 로직 필요.
- **그룹 라벨**: 그룹 자체는 그대로 두고 "순서만" 바꾸는 것이 구현이 단순함. 그룹 이름 변경까지 하면 설정 UI가 복잡해짐.

## 3. 보드·테이블 크기 사용자 지정

### 3.1 요구사항 정리

- **테이블**: 컬럼 너비·컬럼 순서는 이미 가능. 여기서 말하는 "크기"는 예를 들어 (1) 테이블이 들어 있는 **패널 높이**, (2) 테이블과 다른 카드/보드 영역의 **비율**.
- **보드/카드**: 대시보드·나의 대시보드 등에서 Card/Row/Col로 고정된 레이아웃을, 사용자가 **영역 구분선을 드래그해 비율 조정**할 수 있게 하는 것.

### 3.2 기술적 가능 여부

- **가능함.**  
  - **패널 비율**: 상/하 또는 좌/우로 나뉜 구역에 **리사이저블 패널**을 쓰면, 사용자가 구분선을 드래그해 비율을 바꾸고, 그 비율을 저장해 두었다가 다음 접속 시 복원할 수 있음.  
  - **테이블 높이**: 테이블을 감싼 패널의 높이를 사용자 조절 가능하게 하면 됨 (리사이저블 패널의 한 축).

### 3.3 구현 방식 제안

| 대상 | 방식 | 저장 | 비고 |
|------|------|------|------|
| **페이지 내 상/하 패널** (예: 상단 통계 카드 vs 하단 테이블) | `react-resizable-panels` 등으로 상·하 패널 분할, 드래그로 비율 변경 | localStorage `layout_panel_ratio_{pageId}_{panelKey}` (예: `dashboard_main`) | 각 페이지별로 패널 키 설계 |
| **좌/우 패널** (예: 나의 대시보드에서 좌측 필터·우측 목록) | 동일 라이브러리로 좌·우 분할 | 동일 | |
| **테이블 높이** | 테이블을 감싼 컨테이너를 리사이저블 패널의 자식으로 두고, 패널 높이 = 테이블 영역 높이 | 위와 동일한 비율 저장으로 간접 저장 | 테이블 자체는 scroll 영역으로 두면 됨 |
| **카드 그리드 크기** | 각 카드의 width/height를 사용자 지정해 그리드에 반영 | 복잡. 카드별 id + 크기 저장, 그리드 레이아웃 라이브러리 필요 | 우선순위 낮음. 먼저 패널 비율만 적용 권장 |

- **권장 라이브러리**: [react-resizable-panels](https://github.com/bvaughn/react-resizable-panels) — PanelGroup + Panel + 드래그 핸들, 방향(가로/세로), defaultSize·minSize 등 지원. `onLayout`으로 비율 저장 가능.

### 3.4 저장 구조 제안

- **키**: `layout_panel_ratio_{strPageId}_{strPanelGroupId}`.
- **값**: `number[]` — 각 Panel의 defaultSize 비율, 예: `[30, 70]` (상 30%, 하 70%).
- **적용**: 해당 페이지 마운트 시 저장된 비율이 있으면 PanelGroup의 defaultSize로 적용, 없으면 코드 기본값 사용.

### 3.5 적용 대상 페이지 예시

| 페이지 | 적용 후보 | 비고 |
|--------|------------|------|
| DashboardPage | 상(통계 카드) / 하(프로덕트 테이블) | 한 개의 세로 구분선 |
| MyDashboardPage | 상(탭·필터·통계) / 하(테이블 또는 카드 목록) | 한 개의 세로 구분선 |
| QueryPage | 스텝 카드 vs 결과 영역 등 | 필요 시 구분선 추가 |

- 테이블 **컬럼** 너비/순서는 이미 AppTable에서 테이블별로 저장되므로, "테이블 크기"는 위와 같이 **테이블이 들어 있는 패널의 크기(비율)** 로 두는 것이 일관적임.

## 4. 공통 저장·동기화

- **저장소**: 현재처럼 **localStorage** 우선 적용 가능. 키 접두사 예: `db-event-manager-layout`, `db-event-manager-theme`(기존).
- **기기/브라우저 간 동기화**: 나중에 필요하면 백엔드에 사용자별 레이아웃 설정 API를 두고, 로그인 시 불러와서 localStorage에 merge하는 방식으로 확장 가능. (현재 테마는 로컬만.)

---

## 5. 구현 난이도·우선순위 요약

| 기능 | 난이도 | 비고 |
|------|--------|------|
| **메뉴 순서(설정에서 변경)** | 중 | 저장 구조 + 설정 UI(리스트 + 위/아래 버튼 또는 간단 드래그). MainLayout에서 순서 적용 로직. |
| **메뉴 순서(사이드바 드래그)** | 중상 | Sortable 메뉴 또는 커스텀 메뉴 렌더링. 그룹/항목 계층 처리. |
| **패널 비율(보드/테이블 영역)** | 중 | react-resizable-panels 도입, 페이지별 PanelGroup 구성, 비율 저장/복원. |
| **카드 단위 크기** | 상 | 그리드 레이아웃 + 카드별 크기 저장. 필요 시 나중에. |

- **우선순위 제안**: (1) 메뉴 순서(설정 UI) → (2) 대시보드/나의 대시보드 패널 비율 → (3) 그 외 페이지 패널·카드 크기는 요구에 따라 단계적 적용.

---

## 6. 요약

- **가능 여부**: 메뉴 위치(순서)와 보드/테이블(패널) 크기 모두 **구현 가능**.
- **이미 있는 것**: 사이드바 너비, 테이블 컬럼 순서·너비는 이미 사용자 지정 + localStorage 저장됨.
- **추가 시**:  
  - **메뉴**: 그룹/항목 순서를 배열로 저장하고, MainLayout에서 해당 순서로 렌더링. 설정 드로어에서 순서 변경 UI 제공.  
  - **보드/테이블**: 리사이저블 패널로 영역 비율을 드래그해 조정하고, 비율을 페이지·패널별로 저장해 복원.

이 문서는 검토용 제안이며, 실제 반영 시 SPEC.md·디자인 시스템과 맞춰 단계별로 적용하면 됩니다.
