# QA/LIVE 반영 요청 버튼 — 게이지 전환 UX 검토

## 요구 요약

- **QA 반영 요청** / **LIVE 반영 요청** 버튼 하나만 두고, 재요청은 같은 버튼에서 게이지로 전환 후 실행.
- 방안 1: 버튼 위에 **마우스 포커스(호버)** 시 일정 시간 동안 게이지가 차고, 다 차면 재요청 버튼으로 변경 → 클릭 시 재요청.
- 방안 2: 버튼을 **누르고 있는 동안** 게이지가 차고, 다 차면 재요청 모드로 변경 → 클릭(또는 놓기) 시 재요청.

---

## 적용 위치

| 상태         | 현재 버튼 구성                    | 통합 후 동작 |
|--------------|-----------------------------------|--------------|
| `qa_verified` | LIVE반영 요청 + QA 재반영 요청    | LIVE 반영 요청 1개. 기본 클릭 = LIVE 요청. 게이지 완료 후 클릭 = QA 재반영 요청. |
| `live_verified` | LIVE 재반영 요청만 표시           | 완료 후에는 “진행” 버튼이 없어서, 기존처럼 재요청 버튼 단독 유지하거나, “LIVE 반영 요청” 스타일로 보이게 한 뒤 게이지로 재요청 전환 가능. |

---

## 기술적 가능 여부

### 둘 다 구현 가능

- **방안 1 (호버)**  
  - `onMouseEnter`에서 `setInterval` 또는 `requestAnimationFrame`으로 진행률(0→100%) 증가.  
  - `onMouseLeave`에서 타이머 정지 및 진행률 리셋.  
  - 100% 도달 시 버튼 라벨/아이콘을 “재요청”으로 바꾸고, `onClick`에서 `isReRequestMode`면 재요청 API 호출.

- **방안 2 (롱프레스)**  
  - `onPointerDown`에서 타이머 시작, `onPointerUp` / `onPointerLeave` / `onPointerCancel`에서 타이머 정지 및 리셋.  
  - 게이지 100% 시 재요청 모드로 전환, 이후 클릭 시 재요청.  
  - 터치도 `pointer` 이벤트로 동일하게 처리 가능.

---

## 방안 비교

| 항목           | 호버(방안 1)              | 롱프레스(방안 2)           |
|----------------|---------------------------|----------------------------|
| 데스크톱       | 직관적. 실수로 재요청 가능성 있음. | 의도적 동작에 가깝이 제어 가능. |
| 터치/모바일    | 호버 없음 → 게이지 불가.  | 롱프레스로 동일 UX 제공 가능. |
| 접근성         | 키보드 포커스만으로는 “호버 시간” 재현이 애매. | 포커스 유지 후 키 길게 누르기로 유사 동작 가능. |
| 실수 방지      | 마우스만 올려도 차서 오클릭 위험. | “꾹 누르기”가 필요해 재요청 오클릭 감소. |

**권장**: **방안 2(롱프레스)** 쪽이 터치·실수 방지·의도 명확성 측면에서 유리.  
데스크톱만 고려하고 “빠르게 재요청”을 강조하면 방안 1도 선택 가능.

## 구현 시 공통 사항

1. **공용 컴포넌트**  
   - 예: `RequestWithRerequestButton`  
   - props: `label`, `rerequestLabel`, `onRequest`, `onRerequest`, `durationMs`, `variant: 'hover' | 'longpress'`.

2. **게이지 표현**  
   - Ant Design `Progress` (line, `percent`, `showInfo={false}`)를 버튼 하단 또는 배경으로 두거나,  
   - `::after` 등으로 버튼 내부에 채워지는 바를 구현.

3. **상태**  
   - `progress: 0..100`, `isReRequestMode: boolean`.  
   - 게이지 100%가 되면 `isReRequestMode = true`, 클릭 시 `onRerequest` 호출 후 리셋.  
   - `mouseLeave` / `pointerUp` 등에서 진행 중이면 `progress`와 `isReRequestMode` 리셋.

4. **접근성**  
   - “일반 클릭: LIVE 반영 요청, 길게 누르면 재요청으로 전환” 같은 설명을 `title`/`aria-label` 또는 툴팁으로 제공.  
   - 재요청 모드일 때 `aria-label`을 “QA 재반영 요청” 등으로 변경.

5. **live_verified**  
   - 여기서는 “LIVE 반영 요청”이 없고 “LIVE 재반영 요청”만 있으므로,  
     - 그대로 단일 재요청 버튼으로 두거나,  
     - “LIVE 반영 요청” 스타일 라벨로 보이게 하여 롱프레스 시 “재요청” 확인 후 실행하는 식으로 통일할 수 있음.

---

## 결론

- **가능 여부**: 두 방안 모두 구현 가능.  
- **추천**: 터치·실수 방지·의도 명확성을 위해 **롱프레스(방안 2)** 우선 검토.  
- **적용 범위**: `qa_verified`에서 LIVE 요청 + QA 재반영 요청을 한 버튼으로 통합하는 것부터 적용하고, `live_verified`는 필요 시 동일 패턴으로 확장.

이 문서는 검토용이며, 구현 시 `MyDashboardPage.tsx`의 해당 버튼 영역을 `RequestWithRerequestButton`으로 교체하면 됨.
