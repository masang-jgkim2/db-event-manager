# 이벤트 메뉴·화면 구성 검토

목표: 이벤트 메뉴 고도화, 다중 쿼리 실행 고도화 전에 현재 구성을 정리하고 필요/불필요를 검토한다.

---

## 1. 현재 메뉴·화면 구조

### 이벤트 그룹 (권한: dashboard / product / event_template / db_connection)

| 메뉴 | 경로 | 페이지 | 역할 |
|------|------|--------|------|
| 대시보드 | `/` | DashboardPage | 프로덕트·이벤트 템플릿·생성된 이벤트 수 카드, 프로덕트 현황 테이블 |
| 프로덕트 | `/products` | ProductPage | 프로덕트(게임/서비스) CRUD, 서비스(국내/해외) 관리 |
| 이벤트 템플릿 | `/events` | EventPage | 이벤트 템플릿 CRUD, 단일/다중 쿼리 세트, 치환자 템플릿 |
| DB 접속 정보 | `/db-connections` | DbConnectionPage | MSSQL/MySQL 접속 CRUD, 환경(QA/LIVE), 종류(GAME/WEB/LOG) |

### 운영 그룹 (권한: my_dashboard / instance)

| 메뉴 | 경로 | 페이지 | 역할 |
|------|------|--------|------|
| 나의 대시보드 | `/my-dashboard` | MyDashboardPage | 이벤트 인스턴스 목록, 반영(DEV/QA/LIVE), 상세·수정·쿼리수정·컨펌·실행·확인·재요청 |
| 이벤트 생성 | `/query` | QueryPage | 프로덕트→국내/해외→이벤트 선택→입력→반영 범위→생성 |

### 사용자 그룹 (권한: user / role)

| 메뉴 | 경로 | 페이지 | 역할 |
|------|------|--------|------|
| 사용자 | `/users` | UserPage | 사용자 CRUD, 역할 배정 |
| 역할 권한 | `/roles` | RolePage | 역할 CRUD, 권한 세분화 |

---

## 2. 이벤트 플로우 관점 정리

- **설정(관리)**  
  프로덕트 → 이벤트 템플릿 → DB 접속 정보  
  (이벤트 템플릿이 어떤 프로덕트·어떤 DB에 쿼리 날릴지 정의)

- **운영(실무)**  
  이벤트 생성(QueryPage) → 나의 대시보드(상세·수정·컨펌·실행·확인)

- **대시보드(/)**
  - 요약 숫자 + 프로덕트 현황 테이블  
  - “생성된 이벤트”는 현재 0 고정(인스턴스 API 연동 없음)

---

## 3. 필요 vs 조정 가능 항목

### 꼭 유지하는 것이 좋은 것

| 항목 | 이유 |
|------|------|
| **프로덕트** | 이벤트·DB 접속의 단위, 서비스(국내/해외) 정의 |
| **이벤트 템플릿** | 쿼리 세트·치환자·반영 로직의 기준 |
| **DB 접속 정보** | MSSQL/MySQL 실행 대상, 다중 쿼리 고도화의 기반 |
| **이벤트 생성** | 실제 인스턴스 생성 진입점 |
| **나의 대시보드** | 워크플로(DEV→QA→LIVE)와 실행·확인·재요청의 중심 |

### 검토·간소화 후보

| 항목 | 현재 | 제안 |
|------|------|------|
| **대시보드(/)**
  | 프로덕트/이벤트 수 카드 + 프로덕트 테이블. “생성된 이벤트”는 0 고정 | (1) **유지하되** “생성된 이벤트”를 인스턴스 API로 연동해 실제 건수 표시.  
  (2) 또는 대시보드를 **나의 대시보드로 리다이렉트**하고, 관리용 요약은 “프로덕트/이벤트 템플릿” 메뉴 상단에만 두는 방식으로 단순화 가능. |
| **이벤트 그룹 vs 운영 그룹** | 이벤트: 대시보드·프로덕트·이벤트 템플릿·DB접속 / 운영: 나의 대시보드·이벤트 생성 | 그룹명을 **“관리” / “운영”** 등으로 바꾸거나, “이벤트” 그룹에 “이벤트 생성”을 넣어 “이벤트 관련은 한 그룹에” 모을 수 있음. (선호도에 따라 선택) |
| **별도 “대시보드”** | dashboard.view 권한으로 `/` 단독 페이지 | GM/기획자는 나의 대시보드만 쓸 수 있어도 됨. **대시보드를 없애고** 기본 진입을 나의 대시보드(`/my-dashboard`)로 통일하면 메뉴 수가 줄어듦. |

### 없어도 되는 것 (선택적 제거)

- **대시보드(/) 페이지 자체**  
  - 역할: 요약 숫자 + 프로덕트 테이블.  
  - 프로덕트 테이블은 “프로덕트” 메뉴에 이미 있음.  
  - **제거 시**: `/`를 나의 대시보드 또는 이벤트 생성으로 리다이렉트, “이벤트” 그룹에서 “대시보드” 메뉴만 제거하면 됨.

### 고도화 시 함께 보면 좋은 것

- **이벤트 템플릿(EventPage)**  
  - 다중 쿼리 세트 UI(탭, 세트 추가), 기본값/치환자 안내 유지.  
  - 다중 쿼리 실행 고도화(MSSQL/MySQL)와 맞추어 “실행 순서”, “에러 시 다음 세트 중단 여부” 등 옵션 확장 검토.

- **나의 대시보드(MyDashboardPage)**  
  - 반영(DEV/QA/LIVE), 반영 범위, 다중 쿼리 세트 탭 표시는 유지.  
  - 실행 결과를 “세트별”로 보여주는 등 다중 쿼리 실행 고도화와 연동.

- **이벤트 생성(QueryPage)**  
  - 반영 범위, 다중 세트 입력/탭은 유지.  
  - 생성 후 나의 대시보드로 이동하는 흐름 유지.

---

## 4. 권한과 메뉴 노출

- **dashboard.view** → 대시보드(/) 메뉴  
- **product.view** → 프로덕트  
- **event_template.view** → 이벤트 템플릿  
- **db_connection.view** 또는 **db.manage** → DB 접속 정보  
- **my_dashboard.view** → 나의 대시보드  
- **instance.view** 또는 **instance.create** → 이벤트 생성  

대시보드(/)를 제거하면 `dashboard.view`는 “첫 진입 페이지 결정” 용도로만 쓰이거나, 제거 후 기본 진입을 `my_dashboard.view` 기준으로 통일할 수 있음.

## 5. 정리 제안 (요약)

| 구분 | 제안 |
|------|------|
| **유지** | 프로덕트, 이벤트 템플릿, DB 접속 정보, 이벤트 생성, 나의 대시보드, 사용자, 역할 권한 |
| **검토** | 대시보드(/) — 인스턴스 건수 연동 후 유지 vs 제거 후 진입을 나의 대시보드로 통일 |
| **선택** | 이벤트 그룹에 “이벤트 생성” 포함 여부, 그룹명(이벤트/관리/운영) 정리 |

다음 단계: 다중 쿼리 실행 고도화(MSSQL/MySQL) 시 **이벤트 템플릿·이벤트 생성·나의 대시보드**의 “쿼리 세트” 정의와 실행 결과 표시를 일치시키면 된다.
